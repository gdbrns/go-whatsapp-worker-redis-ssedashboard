System Overview

System Purpose

The Go-WhatsApp Worker is a high-performance, scalable system designed to process and send WhatsApp messages through a queue-based architecture. It provides a robust solution for handling large volumes of messages with built-in retry mechanisms, rate limiting, and real-time monitoring.

Key Features

  • Queue-based Processing: Uses Redis Streams as a message queue for reliable message handling
  • Campaign System: Advanced bulk messaging with campaign and chunk management for large-scale operations
  • Rate Limiting: Dual-profile rate limiting (manual/bulk) to comply with WhatsApp API limits
  • Retry Mechanism: Automatic retry with exponential backoff for failed messages
  • Real-time Monitoring: Dashboard with live statistics and health checks
  • Scalable Architecture: Designed for horizontal scaling with concurrent worker pools
  • Database Integration: PostgreSQL for persistent message, campaign, and chunk storage
  • File Lifecycle Management: Temporary uploads are removed after processing with hourly background cleanups

System Architecture

High-Level Architecture

The system follows a clean architecture pattern with distinct layers for different concerns:

[External Systems] 
       ↓ (HTTP API)
[     API Layer     ]
       ↓
[  Message Service  ] ←→ [  Redis Queue   ]
       ↓                      ↓
[ Message Processor ] ←→ [ PostgreSQL DB ]
       ↓
[ WhatsApp Client   ] → [ WhatsApp API  ]
                               ↓
                          [ Recipients   ]

Architecture Layers

  1. Interface Layer: REST API endpoints and web dashboard for user interaction
  2. Service Layer: Business logic for message handling and processing
  3. Worker Layer: Core message processing logic with rate limiting and retry mechanisms
  4. Integration Layer: Communication with Redis, PostgreSQL, and WhatsApp API

Data Flow

  1. Messages are received via API endpoints and stored in the Redis queue (media files persisted in tmp/)
  2. Worker processes continuously poll Redis for new messages
  3. Messages are retrieved, validated, and sent to WhatsApp API
  4. Results are stored in PostgreSQL database for tracking
  5. Temporary media files are deleted after success or terminal failure, with hourly background sweeps for leftovers
  6. Dashboard provides real-time statistics via Server-Sent Events

Worker Logic

Message Processing Flow

The worker follows a continuous loop to process messages from the Redis queue:

  1. Queue Polling: Workers claim entries from Redis Streams via XREADGROUP with a blocking timeout.
  2. Message Retrieval: Deserialize the JSON payload and hydrate message + campaign context.
  3. Status Update: Mark the message row as processing in PostgreSQL before calling the sender.
  4. Rate Limiting: Apply profile-driven pacing (manual vs bulk) before each outbound request.
  5. Message Sending: Invoke the WhatsApp client with typing simulation, media duplication, and presence toggles.
  6. Result Persistence: Persist API response data, update status, and acknowledge the stream entry.
  7. Retry Handling: Schedule exponential backoff via the delayed ZSET for retryable failures up to WORKER_MAX_RETRIES.
  8. File Cleanup: Delete temporary media on success or terminal failure and rely on hourly sweeps for leftovers.

Rate Limiting Implementation

The system implements sophisticated rate limiting to comply with WhatsApp API guidelines:

  • Profile-aware Delays: Manual mode uses RATE_LIMIT_DELAY_MIN/MAX; bulk mode derives delays from RATE_DELAY_MEAN_MS ± RATE_DELAY_JITTER_MS.
  • Typing Simulation: Random typing windows (manual = configured min/max, bulk = 3-8s) combined with presence updates.
  • Natural Breaks & Cooldowns: Manual profile uses NATURAL_BREAK_INTERVAL and duration bounds; bulk profile enforces cooldowns after RATE_BURST_LIMIT sends using RATE_COOLDOWN_SECONDS.
  • Randomization: All delays include jitter to avoid deterministic patterns.

Error Handling & Retry Logic

Robust error handling with automatic retry mechanisms:

  • Incremental Retries: Messages can be retried up to configured maximum attempts
  • Backoff Strategy: Exponential backoff for repeated failures
  • Stuck Message Recovery: Automatic recovery of messages stuck in processing state
  • Error Classification: Different handling for temporary vs permanent failures

Health Monitoring

Continuous health monitoring ensures system reliability:

  • Component Health Checks: Regular checks for Redis, PostgreSQL, and WhatsApp API
  • Performance Metrics: Real-time collection of processing statistics

Redis Logic

Queue Management

Redis serves as the primary message queue with the following characteristics:

  • Data Structure: Uses Redis Streams (XADD/XREADGROUP) with consumer groups for ordered processing
  • Blocking Operations: Leverages XREADGROUP with blocking semantics for efficient queue polling
  • Message Serialization: JSON encoding for message data
  • Queue Statistics: Real-time monitoring of stream depth and consumer performance

PostgreSQL table structure for message storage:

SQL 
CREATE TABLE outgoing_messages (
    id VARCHAR PRIMARY KEY,
    phone VARCHAR NOT NULL,
    message TEXT NOT NULL,
    message_type VARCHAR NOT NULL,
    caption TEXT,
    file_path VARCHAR,
    image_url VARCHAR,
    view_once BOOLEAN DEFAULT FALSE,
    compress BOOLEAN DEFAULT FALSE,
    is_forwarded BOOLEAN DEFAULT FALSE,
    reply_message_id VARCHAR,
    duration INTEGER DEFAULT 0,
    provider VARCHAR DEFAULT 'go-whatsapp',
    campaign_id VARCHAR,
    chunk_id VARCHAR,
    queue_status VARCHAR DEFAULT 'pending',
    status VARCHAR DEFAULT 'pending',
    attempts INTEGER DEFAULT 0,
    max_attempts INTEGER DEFAULT 3,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    sent_at TIMESTAMP WITH TIME ZONE,
    message_id VARCHAR,
    api_response JSONB,
    error_message TEXT
);

CREATE TABLE campaigns (
    id VARCHAR PRIMARY KEY,
    token VARCHAR UNIQUE NOT NULL,
    status VARCHAR DEFAULT 'pending',
    message_type VARCHAR NOT NULL,
    total_chunks INTEGER DEFAULT 0,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE chunks (
    id VARCHAR PRIMARY KEY,
    campaign_id VARCHAR NOT NULL REFERENCES campaigns(id),
    recipients JSONB NOT NULL,
    status VARCHAR DEFAULT 'pending',
    attempts INTEGER DEFAULT 0,
    max_attempts INTEGER DEFAULT 3,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    error_message TEXT
);
